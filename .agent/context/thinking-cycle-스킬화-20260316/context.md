# 컨텍스트

## 배경

사용자(경수님)가 Thinking Cycle이 코드 작업에서만 발동되고, 학습 질문이나 상태 확인 등 비-코드 인터랙션에서는 누락되는 문제를 발견.

근본 원인: TC가 각 도메인 스킬의 execution-protocol.md에 임베딩(Step -1)되어 있어서, 스킬 매칭이 안 되면 TC도 발동되지 않음.

## 결정사항

| 결정 | 선택 | 이유 |
|------|------|------|
| 해결 방향 | B: 스킬 구조 변경 | antigravity.md 강화(A)는 프롬프팅 의존으로 일관성 보장 불가 |
| TC 스킬 유형 | 최우선 + 비차단 | 모든 인터랙션에서 먼저 로드, 도메인 스킬 매칭 방해 금지 |
| 기존 Step -1 처리 | 제거 (A) | SessionStart 훅으로 구조적 강제하므로 이중 안전망 불필요, DRY 원칙 |
| 강제 메커니즘 | SessionStart 훅 | 프롬프팅이 아닌 구조적 강제로 누락 방지 |

## 제약

- thinking-cycle.md 내용(프로토콜 자체)은 변경하지 않음
- 도메인 스킬의 핵심 실행 로직(Step 0~N)은 변경하지 않음
- verify-*, manage-skills는 TC 미적용 (검증 스킬은 별도 프로토콜)

## 리스크

| 리스크 | 완화 |
|--------|------|
| TC 스킬이 로드 안 되면 전체 TC 누락 | SessionStart 훅으로 구조적 강제 |
| 도메인 스킬과 TC 스킬의 중복/충돌 | TC 스킬은 프로토콜만, 도메인 스킬은 실행만 담당하도록 역할 분리 |
| 12개 파일 일괄 수정의 리그레션 | 제거할 섹션이 동일 패턴이므로 일괄 처리 가능 |
