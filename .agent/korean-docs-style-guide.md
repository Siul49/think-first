# 한국어 문서 스타일 가이드

## 목적

프로젝트 문서를 한국어 우선 체계로 유지하기 위한 기준서.
문서를 읽는 사용자가 외부 도움 없이 구조를 이해하고, 수정 우선순위와 운영 방향을 스스로 판단할 수 있도록 돕는다.

## 적용 범위

- `antigravity.md` (프로젝트 설정)
- `.agent/commands/*.md` (슬래시 커맨드)
- `.agent/skills/**/SKILL.md` (레거시 스킬, 참조용)
- `.agent/skills/**/resources/*.md` (레거시 리소스, 참조용)
- `.agent/skills/_shared/*.md` (공유 프로토콜, 참조용)

## 핵심 원칙

1. 제목, 섹션명, 설명, 보고 형식, 예시 설명은 한국어를 기본으로 작성한다.
2. 식별자, skill id, tool id, 파일 경로, 키 이름, 명령어는 원문을 유지한다.
3. 영어는 꼭 필요한 경우에만 괄호 병기 또는 예외 목록으로 남긴다.
4. 문체는 설명형 존댓말을 기본으로 하되, 실행 규칙은 간결한 지시형 문장으로 통일한다.
5. 직역보다 사용자가 바로 이해할 수 있는 자연스러운 한국어를 우선한다.

## 용어 통일 기준

| 기준 용어 | 기본 표기 | 비고 |
|----------|-----------|------|
| workflow | 워크플로우 | 필요할 때만 `(workflow)` 병기 |
| guide | 가이드 | 제목에서는 `가이드`, 일반 문장에서는 `안내` 허용 |
| orchestrate / orchestration | 오케스트레이션 | 기능명일 때 `오케스트레이션`, 일반 설명에서는 `조율` 허용 |
| coordinate / coordination | 조율 | 다중 에이전트 협업 맥락에서 사용 |
| verify | 검증 | 자동/수동 확인 절차를 모두 포함 |
| review | 리뷰 | QA 맥락에서는 `리뷰`, 일반 검토는 `검토` 허용 |
| checklist | 체크리스트 | 점검 항목 문서에 공통 사용 |
| playbook | 대응 지침 | 오류 복구 문서 제목에 사용 |
| reporting | 보고 방식 | `Reporting` 공통 헤더 대체 |
| examples | 예시 | 입력/출력 예시는 `입력`, `결과`로 표기 |
| execution protocol | 실행 절차 | 단계형 작업 흐름 문서에 사용 |
| reasoning | 사고 템플릿 | 내부 추론을 구조화하는 문서에 사용 |

## 영어 잔존 허용 기준

아래 항목은 실행 안정성과 원문 식별을 위해 영어를 유지한다.

- 파일명, 디렉터리명, 경로
- 코드 블록 안의 명령어, 옵션, 식별자, JSON/YAML 키
- 표준/기술 약어: MCP, API, DTO, JWT, REST, CRUD, OWASP, WCAG, SSE, OpenAPI, TypeScript, Tailwind, Next.js, FastAPI, Flutter
- 상태 코드나 결과 값 예시: `blocked`, `quota_exceeded`, `memory_fallback`, `PASS`, `FAIL`

## 유지보수 메모

- 새 문서를 추가할 때는 먼저 이 기준서의 용어 표를 확인한다.
- 검증 스크립트나 런타임 생성물은 번역 대상이 아니다.
- 사용자에게 보이는 이름과 설명을 바꿀 때도 경로와 키 이름은 그대로 둔다.
