## 👤 경수 (본인)

> 💡 데이터 품질과 수집 안정성의 핵심을 담당합니다.

### 1️⃣ 데이터 품질 정리 (DQ-1 ~ DQ-3)

| 항목 | 내용 |
| --- | --- |
| 예상 공수 | 9~12시간 |
| 위치 | Supabase SQL Editor / 로컬 스크립트 |

**왜 해야 하는가?**
LLM 파싱 실패 시 `max_capacity=100` 같은 가짜 값이 저장되어 사용자에게 노출되면 "앱 정보가 엉터리"라는 인식을 줍니다. 서울 외 데이터 제거 및 수기 보정이 필수입니다.

**할 일 요약**
- [ ]  DQ-1: 서울 외 좌표 데이터 정리 (SQL)
- [ ]  DQ-2: `max_capacity=100` 파싱 실패 건 실측 확인 및 수기 보정
- [ ]  DQ-3: 기타 데이터 정합성 점검

### 2️⃣ 수집 파이프라인 실패 격리 (BE-32)

| 항목 | 내용 |
| --- | --- |
| 우선 | **실패 격리** 먼저, 병렬화는 그 다음 |
| 예상 공수 | 5~6시간 |
| 위치 | `app/services/room_collection_service.py` L33~L61 |

**왜 해야 하는가?**
하나의 크롤링 실패가 전체 수집을 중단시키지 않아야 합니다. 로컬(Ollama) 환경에서 안정적으로 리포트를 뽑을 수 있어야 합니다.

**할 일 요약**
- [ ]  개별 수집 실패가 전체를 중단시키지 않도록 `try/except` 격리
