# Phase 2 ~ Phase 4 프로젝트 고도화 백로그 (상세)

본 문서는 백엔드 기능 중 향후 서비스 안정화 및 구조 고도화를 위해 수행해야 할 기능 보강 및 성능 개선 스펙입니다. 각 태스크의 적용 당위성과 기대효과를 상세히 기록합니다.

## Phase 2 (즐겨찾기 완성)
- **BE-11 (Task 2-1)**: `GET /api/favorites/details` 추가 (room join 기반 상세 반환).
  - **당위성**: 현재는 즐겨찾기 목록 조회가 ID 리스트 스칼라 값만 던져주기 때문에, 프론트엔드가 카드를 화면에 직접 그리려면 또 다시 각 ID별로 상세 조회 API를 호출해야 합니다(N+1 렌더링 지연 발생 위험). 이 API를 만들면 가벼운 페이로드 한 번으로 렌더링이 가능해집니다.
- **BE-12 (Task 2-2)**: device당 찜 목록 상한 제한(예: 50개) 정책 및 커스텀 예외 반환.
  - **당위성**: 누군가 악의적 목적으로 혹은 스크립트로 찜하기를 1만개씩 날려 Supabase 용량을 고갈시키거나 DB를 타격하는 것을 방지하는 최소한의 자원 방어 논리입니다. 51번째 추가 시 명확한 400 계열 커스텀 에러코드를 반환하여 UI에 "저장 공간 초과" 팝업을 띄우게 합니다.
- **BE-13 (Task 2-3)**: `validate_device_id` UUID v4 검증 기능 추가.
  - **당위성**: X-Device-Id 헤더가 v4 UUID가 아닌 타임스탬프 기반 v1 등 난해한 식별자, 혹은 단순 랜덤 문자열일 경우 저장소 데이터 무결성이 어긋납니다. 정규표현식이나 UUID 파싱으로 v4만 남기고 잘라내야(normalize) 오염된 데이터로 즐겨찾기 테이블이 망가지는 것을 막을 수 있습니다.
- **BE-14 (Task 2-4)**: 저장소 싱글톤 방식 정리.
  - **당위성**: 매번 라우터에 들어올 때마다 레포지토리, 즉 DB 커넥션이나 인스턴스가 계속 새로 생성되면 메모리 및 커넥션 누수가 발생해 자원을 갉아먹습니다. lru_cache나 모듈 레벨 싱글톤을 도입하여 서버 트래픽이 많을 때를 대비한 자원 절약을 수행합니다.

## Phase 3 (크롤링 안정성)
- **BE-21 (Task 3-1)**: 네이버 등 외부 요청 시 Semaphore 제어 및 429/503 상태코드 backoff retry 로직.
  - **당위성**: 검색이 한 번에 10명만 들어와도 1명당 크롤링 요청이 10~20개가 뿌려집니다. 네이버, 그루브 입장에서는 갑자기 트래픽이 폭증(burst)하는 DDoS 행위로 인지하여 즉각 IP 차단(블랙리스트)을 먹일 수 있습니다. 서서히 분산시켜 쏘고, 터지면 잠깐 쉬었다 다시 시도하는 백오프 재시도 로직이 서비스 생존의 필수 요건입니다.
- **BE-22 (Task 3-2)**: 특정 사이트(Groove 등) 로그인 세션 재사용 및 401/403 시 재인증.
  - **당위성**: 매번 조회할 때마다 로그인을 시도하게 되면 그만큼 시간도 오래 걸리고(타임아웃 유발), 서버 부하가 큽니다. 쿠키나 로그인 세션을 캐싱해두면 응답을 훨씬 안정적으로 빠르게 받을 수 있습니다.
- **BE-23 (Task 3-3)**: `business_id` -> `crawler_type`을 매핑하는 구조의 DB화.
  - **당위성**: 현재는 파이썬 딕셔너리에 수동으로 하드코딩 되어 있습니다. 새로운 합주실 지점이 추가될 때마다 백엔드 개발자가 서버 코드를 직접 열고, 변수를 추가해서, 서버를 껐다 켜야(재배포)만 반영됩니다. DB에 합주실 타입 컬럼을 뚫어서, 어드민에서 체크값 하나만 바꾸면 즉시 작동하도록 유지보수성을 극강으로 높이는 작업입니다.
- **BE-24 (Task 3-4)**: 크롤러의 `max_workers=1` 기술적 의도 주석/문서화.
  - **당위성**: Playwright(브라우저 제어)가 비동기 루프 상에서 충돌이나 교착 상태(Deadlock)를 일으키는 것을 막기 위해 동시성 처리를 제약한 중요한 아키텍처 의도입니다. 향후 다른 개발자가 성능 올리겠다고 스레드 수를 올렸다가 서버가 영구 블로킹(멈춤)되는 끔찍한 사태를 막으려면 왜 1로 했는지 히스토리를 적어두어야 합니다.
- **BE-25 (Task 3-5)**: 공유 `httpx.AsyncClient` limits/timeout 표준화.
  - **당위성**: AsyncClient를 매번 생성하고 닫는 행위는 소켓 리소스를 빠르게 고갈시킵니다. 타임아웃 규칙과 커넥션 한도를 명시적으로 주어 리소스 오버플로우나 유령 커넥션(Zombie Connection) 문제가 안 생기도록 합니다.

## Phase 4 (성능/운영성)
- **BE-31 (Task 4-1)**: 좌표 필터(위도/경도) 처리를 파이썬 레벨이 아닌 DB 레벨(RPC/PostGIS query)로 위임.
  - **당위성**: 향후 전국 수만 개의 합주실이 추가되었을 때, 모든 데이터를 백엔드로 일단 끌고 온 뒤에 파이썬이 하나하나 좌표 계산을 돌리면 서버가 연산량(CPU) 한계로 메모리 에러(OOM)가 납니다. DB 서버 인덱스를 타고 범위 쿼리를 짜는 편이 대규모 데이터에서 100배는 빠르고 응답도 안정적입니다.
- **BE-32 (Task 4-2)**: `RoomCollectionService` 제한적 비동기 병렬 처리/실패 격리.
  - **당위성**: 하나의 크롤링 태스크가 실패했다고 전체 합주실 수집 플로우가 와르르 깨지면 안 됩니다. 청크 단위로 쪼개고 한쪽이 통신 오류가 발생하더라도, 일단 정상 수집된 나머지들의 데이터는 무사히 넘겨주는 격리(Isolation) 아키텍처입니다. 수집 속도와 장애 면역력 두 마리 토끼를 다 잡습니다.
- **BE-33 (Task 4-3)**: 예약 정보(Availability) TTL(Time-To-Live) 단기 캐시 도입.
  - **당위성**: 불과 3초 전에 같은 "사당 합주실"을 14~16시로 조회한 사람이 있는데, 지금 들어온 유저를 위해 무겁게 크롤러를 또 돌릴 필요가 있을까요? 예약의 성격상 분 단위로 확 바뀌진 않으므로, 1~3분 정도 단기로 레디스(Redis) 캐시 등에 태워주면 API 부하를 획기적으로 줄이고 조회 속도는 0.1초 컷으로 당겨집니다.
- **BE-34 (Task 4-4)**: `/health` 확장 모니터링 및 크롤러 실패율 메트릭(Metric).
  - **당위성**: 네이버 크롤링이 봇 감지로 차단당해 실패율이 90% 이상 치솟는 장애 상황이 생겨도, 외부 모니터링에서는 HTTP 200이 반환되면 서버는 건강하다고 착각합니다. 크롤링 자체의 헬스체크 및 크롤러 실패 지표(Metric)를 꽂아 넣어, 그루브나 드림 등 타사 시스템이 바뀐 경우 즉각 백엔드 모니터에 알람이 울리도록 가시성(Observability)을 확보합니다.
