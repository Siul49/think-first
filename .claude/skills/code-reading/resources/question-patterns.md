# Code Reading - 질문 패턴

## 사전 점검 질문 (Phase 0 통합)

### 코드 수정 명령 시

```
🔍 이 코드를 수정하기 전에 하나 확인할게요:
→ 이 {함수/모듈}이 현재 어떻게 동작하는지 설명해볼 수 있어요?
```

### 이해도 부족 신호 감지 시

| 신호 | 질문 |
|------|------|
| 존재하지 않는 함수/변수 언급 | "혹시 {X}가 어디에 정의되어 있는지 확인해봤어?" |
| 코드 흐름과 맞지 않는 명령 | "현재 데이터 흐름이 A→B→C인데, 경수님이 말한 건 B→A인 것 같은데 어떤 부분을 보고 판단했어?" |
| 모호한 위치 지정 | "어떤 파일의 어떤 함수를 말하는 건지 좀 더 구체적으로 알려줄 수 있어?" |
| 사이드이펙트 미인지 | "이 부분을 바꾸면 {관련 코드}에도 영향이 가는데, 그 부분은 고려했어?" |

### 이해 기반 명령 판단 기준

아래에 해당하면 사전 점검 생략:
- 사용자가 구체적 파일/함수/라인을 명시
- 변경 이유와 영향 범위를 함께 설명
- 이전 대화에서 해당 코드를 이미 논의

## 사후 점검 질문 (Phase 3: 코드 리딩)

### 도메인별 질문 패턴

#### Backend (API/DB/인증)

**What**:
- "이 엔드포인트가 받는 요청과 돌려주는 응답이 뭐야?"
- "이 쿼리가 DB에서 어떤 데이터를 가져오는지 설명해봐"
- "이 미들웨어가 요청 파이프라인에서 어떤 역할을 해?"

**Why**:
- "왜 이 로직을 Service에 넣었어? Router에 넣으면 안 되는 이유는?"
- "이 인덱스를 왜 추가했어? 없으면 어떤 문제가 생겨?"
- "동기 처리 대신 비동기를 선택한 이유는?"

**응용**:
- "이 API에 페이지네이션을 추가한다면 어디를 수정해야 해?"
- "이 서비스에 캐싱을 적용한다면 어떤 전략을 쓸 거야?"
- "이 엔드포인트에 rate limiting을 걸려면 어디에 어떻게 추가해?"

#### Frontend (UI/컴포넌트)

**What**:
- "이 컴포넌트의 props와 렌더링 결과를 설명해봐"
- "이 상태 변경이 어떤 리렌더링을 발생시키는지 알아?"

**Why**:
- "왜 이 상태를 여기서 관리해? 상위 컴포넌트에서 관리하면 안 돼?"
- "useMemo를 쓴 이유는? 없으면 어떤 일이 벌어져?"

**응용**:
- "이 컴포넌트에 로딩 상태를 추가한다면?"
- "이 폼에 새 필드를 추가하려면 어디를 수정해야 해?"

#### 공통 패턴

**흐름 추적**:
- "사용자가 {행동}을 하면 코드가 어떤 순서로 실행돼?"
- "이 데이터가 {시작점}에서 {끝점}까지 어떻게 흘러가?"

**에러 시나리오**:
- "이 코드에서 예외가 발생하면 어떻게 처리돼?"
- "네트워크 에러가 나면 사용자에게 뭐가 보여?"

**경계 조건**:
- "입력이 빈 값이면 어떻게 되는지 알아?"
- "동시에 두 요청이 들어오면 이 코드가 안전해?"

## 후속 질문 패턴 (답변 깊이별)

### 답변이 정확할 때
- "좋아! 그러면 한 단계 더 — {심화 질문}"
- "맞아. 그런데 만약 {예외 상황}이면?"

### 답변이 부분적일 때
- "{맞는 부분} 맞아. 근데 {빠진 부분}은 어떻게 되는 것 같아?"
- "거의 다 맞는데, {힌트}도 생각해봐"

### 답변이 틀렸을 때
- "음, 그 부분이 좀 다른데 — 코드의 {라인}을 다시 한번 봐봐. 뭐가 보여?"
- "그렇게 동작할 것 같지? 근데 실제로 {힌트}가 있어서 달라. 어떻게 달라질까?"

### 모를 때
- "괜찮아, 같이 봐보자. 이 코드를 위에서부터 읽어보면..."
- [설명 후] "자, 그러면 이 부분이 뭘 하는지 이제 설명할 수 있어?"
