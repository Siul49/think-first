# Document 실행 프로토콜

## Step -1: Thinking Cycle (이해도 점검 + 질문 + 결정)
1. **이해도 사전 점검** — 코드 수정 시 `../code-reading/SKILL.md` 사전 점검 수행
   - 사용자가 수정 대상 코드를 이해하고 있는지 확인
   - 이해 부족 시 설명 요구 → 설명 후 진행
2. **질문** — `../_shared/resources/thinking-cycle.md` Phase 0 수행
   - 작업 복잡도에 맞는 소크라테스 질문 선정
   - 사용자 답변 대기 (답변 전 진행 금지)
3. **결정** — 트레이드오프 존재 시 Phase 1 수행
   - 선택지 제시 + 근거 요구
   - 근거 없는 선택은 재질문

## 1단계: 문서화 범위 파악

```
체크리스트:
□ 문서 유형 결정 (API / 아키텍처 / 가이드 / 코드 / 변경)
□ 대상 코드/모듈 확인
□ 기존 문서 유무 확인 (신규 생성 vs 갱신)
□ 대상 독자 확인 (개발자 / 사용자 / 팀원)
□ 프로젝트 문서 스타일 확인
```

## 2단계: 코드 분석

1. 대상 코드 읽기
   - 함수 시그니처, 타입, 파라미터
   - 주석, docstring, JSDoc
   - export/public API 식별
2. 의존성 추적
   - import/require 관계
   - 데이터 흐름
3. 테스트 코드 확인
   - 사용 패턴, 엣지 케이스
   - 기대 동작

## 3단계: 문서 작성

### 신규 생성
1. `doc-templates.md`에서 적절한 템플릿 선택
2. 코드 분석 결과로 내용 채우기
3. 실행 가능한 예시 코드 포함
4. 프로젝트 한국어 문서 기준 적용

### 기존 갱신
1. 기존 문서 전체 읽기
2. 변경된 코드와 문서 비교
3. 변경 부분만 업데이트 (surgical diff)
4. 기존 톤/스타일 유지

## 4단계: 검증

```
검증 체크리스트:
□ 문서 내용이 실제 코드와 일치하는가?
□ 예시 코드가 실제 동작 가능한가?
□ 파일 경로, 링크가 유효한가?
□ 한국어 문서 기준을 준수하는가? (식별자는 영어)
□ 불필요한 중복이 없는가?
```

## Pre-Final Step: 코드 리딩 (Thinking Cycle Phase 3)
1. `../code-reading/SKILL.md` 사후 점검 수행
2. 변경된 코드에 대해 레벨 C(기본) 질문 → 사용자 답변 평가
3. 코드 변경이 없는 작업은 생략

## Final Step: 회고 (Thinking Cycle Phase 4)
1. `../_shared/resources/thinking-cycle.md` Phase 4 수행
2. 회고 질문 제시 → 사용자 답변을 `.claude/reflections/YYYY-MM-DD.md`에 기록
3. 회고 없이 작업 종료 금지
